Tipos de Requisitos

Funcionais (RF)

O que o sistema FAZ.

São as funcionalidades, comportamentos e serviços que o sistema deve fornecer.

Exemplos:
  • O sistema deve permitir login via Google.
  • O usuário pode cadastrar produtos com fotos.
  • O sistema deve gerar relatórios em PDF.

Não-Funcionais (RNF)

Como o sistema É.

São restrições, qualidades e condições sob as quais o sistema deve operar.

Exemplos:
  • Performance: Carregar em < 2s.
  • Segurança: Senhas criptografadas (bcrypt).
  • Disponibilidade: 99.9% uptime.

User Stories: O Padrão Ágil

Como descrever requisitos focando no valor para o usuário.

1

Como um <papel>

2

Eu quero <funcionalidade>

3

Para que <benefício/valor>

Exemplo Real:

"Como um Gerente de Vendas, eu quero visualizar um gráfico de vendas mensais, para que eu possa identificar tendências e cobrar a equipe."

Quando está "Pronto"? (Critérios de Aceite)

Não adianta pedir "um gráfico", precisa detalhar as condições.

Checklist de Aceitação

Funcionalidade: Login do Usuário
  • O usuário deve informar e-mail e senha.
  • Se a senha for inválida, exibir mensagem "Credenciais inválidas" (não dizer qual errou).
  • O botão de "Entrar" deve ficar desabilitado enquanto os campos estiverem vazios.
  • Ao logar com sucesso, redirecionar para a Dashboard.

Priorização: Método MoSCoW

Não dá para fazer tudo. O que é PRIORIDADE?

M

Must Have

Tem que ter. Sem isso, o produto não funciona ou não tem valor. É o MVP.

Ex: Login, Cadastro de Pedido.

S

Should Have

Deveria ter. É importante, mas se faltar, existe um workaround doloroso.

Ex: Recuperação de Senha automática, Exportar PDF.

C

Could Have

Poderia ter. Seria legal ("Nice to have"), mas só se sobrar tempo.

Ex: Dark Mode, Animações, Foto de Perfil.

W

Won't Have

Não terá (agora). Definido que NÃO será feito nesta versão para focar no resto.

Ex: App Nativo (será só Web), Integração com SAP.

O Que é o MVP do TCC1?

Minimum Viable Product

O MVP não é um produto "pela metade" ou "mal feito".

É a menor versão do produto que resolve o problema central do usuário (Core Value) e que funciona de ponta a ponta.

Regra de Ouro para o Semestre

Foque apenas nos requisitos MUST HAVE e alguns SHOULD HAVE.

Melhor um MVP simples que funciona 100%, do que um sistema complexo cheio de bugs.

A Entrega: Proposta de Projeto

O documento oficial que define o seu TCC. (Template LaTeX)

1. Introdução e Problema

Descreva o contexto, o problema que você identificou e a justificativa (por que resolver?). Use os dados da pesquisa bibliométrica aqui!

2. Objetivos

Geral: O que você vai entregar? (Ex: Desenvolver um sistema web para...)

Específicos: Passos para chegar lá (Levantar requisitos, modelar banco, desenvolver frontend...).

3. Escopo (Requisitos)

Liste os Requisitos Funcionais (User Stories principais) e Não-Funcionais. Defina claramente o que o sistema fará.

4. Tecnologias

Stack tecnológica: Linguagens (Python, JS), Frameworks (React, Django), Banco de Dados (PostgreSQL), Ferramentas (Git, Docker).

Atividade Prática - Mão na Massa!

Rascunho da Proposta

Vamos definir o "coração" do seu projeto agora.

  1. Objetivo Geral
    Escreva 1 frase resumindo o projeto ("Desenvolver um sistema X para resolver Y...").
  2. 5 Requisitos Funcionais (User Stories)
    Escreva as 5 funcionalidades principais no formato "Como , quero , para ".
  3. 3 Requisitos Não-Funcionais
    Defina performance, segurança ou disponibilidade.
  4. Definição do MVP
    Marque nos requisitos acima o que é MUST HAVE (obrigatório para a banca).
